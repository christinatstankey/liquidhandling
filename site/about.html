About

The problem

Safety Data Sheets tell you a reagent is flammable, irritating, or hazardous. They do not tell your liquid handler to pre-wet tips three times before aspirating ethanol, or to warm DMSO to room temperature before it becomes a solid plug in your tip. They do not tell it that proteins need LoBind tubes or that DTT is useless if it was made yesterday.

That knowledge lives in the heads of experienced bench scientists. It is passed on verbally, buried in methods sections, or learned by ruining an experiment. The gap between "safe to handle" and "correct to pipette" is where liquid handlers fail — not because the robot lacks precision, but because no one gave it the tacit knowledge the operator takes for granted.

What this does

This database is a transparent rules layer between chemical properties and robot instructions. Each reagent has a structured property profile — vapor pressure, viscosity, light sensitivity, adsorption tendency, and so on. A declarative rules engine maps those properties to handling instructions. Every rule cites a property value or a published source. Nothing is hidden inside a model.

The output is a JSON handling profile a liquid handler can consume directly:

  • Aspiration speed and technique (reverse pipetting, pre-wet cycles, wide-bore tips)
  • Container type (LoBind, amber, flammable cabinet)
  • Preparation requirements (make fresh, warm to RT, thaw method)
  • Safety controls (fume hood, gloves, waste stream, segregation)

Each instruction carries a confidence level derived from the evidence chain — high for SDS-extracted or ontology-grounded facts, lower for bench consensus. The robot knows which instructions to trust.

Data pipeline

Each reagent record is populated by a layered extraction pipeline. Source types are explicitly tracked so the downstream consumer can evaluate reliability:

Source typeWhat it providesConfidence
sds_phrase Literal phrase match in SDS §7/§10 (e.g., "hygroscopic", "flammable") high
ghs_hcode GHS hazard code from SDS §2 (e.g., H228 → flammable solid) high
storage_class ADR storage class from SDS §7 (e.g., 3 → flammable liquid) high
chebi ChEBI ontology role (e.g., CHEBI:25944 → fluorophore → light-sensitive) high
uniprot Swiss-Prot reviewed entry confirms protein identity; LoBind requirement is a handling inference, not a UniProt claim high
manufacturer_protocol NEB product datasheet (storage conditions, glycerol pct, enzyme class) medium
claude_inference Model inference from chemical knowledge — fallback when no other source applies low

What it isn't

This is not an SDS database, not an EH&S compliance tool, and not a general protocol library. It is a narrow, high-fidelity bridge between physicochemical properties and robot-actionable handling decisions, scoped to reagents on a stockroom shelf.

Out of scope by design: cell culture handling (biologically rather than chemically constrained), buffer preparation workflows, and freeform protocol text. These are deferral calls, not omissions — the reagent core needs to be solid before layering on biological or procedural complexity.

Honesty by construction

Rules that rely on common bench practice without a clean primary source are flagged explicitly as field consensus in the UI. Confidence levels are computed from evidence and stored in the record — not re-derived at read time, not hidden. When two sources disagree, the disagreement is kept in the record rather than silently resolved. The goal is to surface uncertainty, not hide it behind a clean output.

The rules engine is a plain YAML file. Anyone can audit it, add to it, or fork it for a different compound class. The database is JSON files checked into version control. No opaque model weights; no hidden heuristics.

Data sourced from manufacturer SDS sheets and published bench protocols. Rules engine: transparent YAML — no ML.
